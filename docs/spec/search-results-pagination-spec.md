# 検索結果ページング改修 仕様ドラフト

対象: Claude Code review 用の仕様ドラフト。
目的: 既存の検索機能を追加機能ではなく「検索画面としての完成度向上」として改修する。

この文書はまだ Codex 実装 handoff ではない。Claude Code が内容を確認し、必要なら ADR / roadmap / handoff に分解する。

## 背景

現状の `search.php` は、作品検索だけ `もっと見る` で12件ずつ追記する。
ユーザー検索は `LIMIT 30` 固定で、ページングも総件数表示もない。

検索画面は「網羅的に探す」画面なので、タイムライン的な追加読み込みより、現在位置・総件数・ページ移動が分かる UI の方が自然。

## User の意図・こだわり

この改修は「機能を増やしたい」よりも、すでにある検索画面が検索画面として納得できる状態になっているか、という違和感から出ている。

User の判断:

- 検索結果は、タイムラインやおすすめのように流し見る場所ではなく、目的の対象を探す場所。
- そのため、`もっと見る` で下へ継ぎ足すだけでは、現在位置・結果規模・戻りやすさが弱い。
- 「2ページ目に行く」「前のページに戻る」「URL を共有する」「あとで同じ検索状態を開く」ができる方が検索画面らしい。
- 検索結果には対象件数が見えてほしい。何件あるか分からない検索結果は、探す画面として情報が足りない。
- 表示件数は固定ではなく、ユーザーが `10 / 30 / 50` から選べるとよい。ざっと見る人と細かく見る人で使い方が違うため。
- 表示件数を切り替えたとき、直前に見ていた位置を完全に維持するのは難しいが、見失いにくくする配慮は欲しい。
- ただし、ページ境界は `per_page` によって動的に変わるため、アンカー作品を必ず新しいページの先頭に置くことは目標にしない。現実的には「その作品を含むページへ移動し、必要ならスクロール」で十分。
- コメント機能は欲しいが、これは新しい投稿データ・権限・削除/通報などを伴う追加機能。今日中の期日では、既存機能の完成度を上げる改修と分ける。

こだわりの整理:

- トップのおすすめはキュレーションなので固定枠でよい。
- 検索は網羅なので、ページ移動・総件数・表示件数切り替えが欲しい。
- 追加機能より、既存機能の「心残り」を潰してポートフォリオとして説明できる状態にする。

## 対象範囲

対象:

- `public/search.php`
- `public/Script/search.js`
- `public/api/search.php`
- `public/Style/gallery-list.css`
- `src/seed.php`

対象外:

- トップのおすすめ一覧
- 作品詳細・プロフィール・お気に入りの表示仕様
- コメント機能
- DB スキーマ変更
- 認証・CSRF・セッション方針の変更

## 仕様

### 1. 「もっと見る」をページ切り替えに置き換える

検索結果は「もっと見る」追記ではなく、ページ切り替えで表示する。

URL 例:

```text
search.php?q=雨&page=2&per_page=30
search.php?tag=抽象&page=3&per_page=10
search.php?type=users&q=a&page=1&per_page=30
```

要件:

- `page` は 1 始まり。
- ページ移動は URL に反映する。
- ブラウザバック / 再読込 / URL 共有で同じ検索状態を復元できる。
- 空検索でも同じページング仕様を使う。
- `page` が不正値の場合は 1 に丸める。
- `page` が最終ページを超える場合は、API が実際の最終ページに丸めて返す。

UI:

- 検索結果の下部にページ操作を置く。
- 最低限は「前へ」「次へ」でよい。
- 余裕があればページ番号も表示する。
- 現在ページ、総ページ数が分かる表示にする。

表示例:

```text
該当 126 件
30件ずつ表示

前へ  1  2  [3]  4  5  次へ
```

### 2. 総件数を表示する

検索結果見出し付近に総件数を表示する。

表示例:

```text
キーワード「雨」の検索結果
該当 42 件
```

空検索:

```text
すべての作品
該当 128 件
```

ユーザー検索:

```text
ユーザー「a」の検索結果
該当 12 件
```

API は `total` を返す。

### 3. 1ページあたりの表示数を切り替える

検索結果画面で `10 / 30 / 50` から表示件数を選べるようにする。

要件:

- デフォルトは `30`。
- 選択値は `per_page` として URL に残す。
- API 側は `10 / 30 / 50` の allowlist に限定する。
- 任意の外部入力をそのまま `LIMIT` に入れない。
- 不正値は `30` に丸める。

UI:

```text
表示件数 [10 ▼]
表示件数 [30 ▼]
表示件数 [50 ▼]
```

### 4. 表示件数切り替え時の位置維持

表示件数を切り替えるとページ境界が変わるため、同じ `page` を維持しても同じ位置を見られる保証がない。

現実案として、切り替え直前に表示していた先頭カードをフロント側のアンカーとして扱う。

要件:

- 切り替え直前の表示中リストの先頭カード ID を保持する。
- 旧 `page` / `per_page` から先頭カードのおおよその絶対位置 `offset = (page - 1) * per_page` を計算する。
- 新しい `per_page` で `newPage = floor(offset / newPerPage) + 1` を計算し、そのページへ移動する。
- 切り替え後、可能ならそのカードまでスクロールする。
- そのカードを新しいページの先頭に固定することは狙わない。

補足:

- **実装判断（2026-06-30）**：表示件数変更時は `page=1` に戻さず、アンカー優先を正式仕様とする。先頭へ戻ると、いま見ていた作品周辺の文脈を失うため。直前の先頭作品を含むページが最終ページ相当になる場合は、表示件数未満の半端ページへ着地してよい。
- この半端ページ着地は、API の `page=1` が少ないバグではない。URL 直打ち・通常のページ番号クリック・リロード時の `page=1` は、常に検索順の先頭から最大 `per_page` 件を返す。表示件数変更だけが、閲覧位置維持のために「アンカーを含むページ」へ移動する。
- 「必ずページ先頭にする」を要件にすると、番号付きページングではなく `offset` 指定や cursor / anchor-based pagination に寄るため、今回は採用しない。
- アンカー ID は API 契約に含めず、ページ JS の一時状態または URL のフロント専用パラメータとして扱う。API は通常の `page` / `per_page` だけを受け取る。
- 切り替え後のページ内にアンカーが見つからない場合は、通常のページ先頭表示でよい。
- 初回表示・URL 直打ち・通常のページ番号クリックではアンカー計算は不要。

### 5. API 契約

`GET /api/search.php`

入力:

| パラメータ | 内容 |
|---|---|
| `type` | `works` または `users`。省略時は `works` |
| `q` | キーワード / ユーザー検索語 |
| `tag` | タグ検索語 |
| `page` | 1 始まりのページ番号 |
| `per_page` | `10 / 30 / 50` のいずれか |

出力:

```json
{
  "type": "works",
  "results": [],
  "total": 42,
  "page": 2,
  "perPage": 30,
  "totalPages": 2,
  "hasPrev": true,
  "hasNext": false
}
```

補足:

- `totalPages` は `max(1, ceil(total / perPage))` とする。0件時も `page=1`, `totalPages=1` を返す。

### 6. SQL / 安全条件

作品検索の不変条件:

- 公開作品のみ。
- ログイン中は自分の作品を除外。
- 非公開作品を所有者以外に返さない。
- `q` はタイトル / 説明検索。
- `tag` はタグ検索。
- SQL はプリペアドステートメントを使う。
- `LIMIT` / `OFFSET` は allowlist / int 化した値だけを使う。

ユーザー検索:

- 表示名 / GitHub username を検索する。
- 公開作品数・獲得スター集計は公開作品のみを対象にする。

総件数:

- 作品検索は、検索条件と同じ `WHERE` で `COUNT(DISTINCT g.id)` を取る。
- ユーザー検索は検索条件に一致するユーザー数を count する。
- 現在の `PER_PAGE+1` による `hasMore` 判定は不要になる。

並び順:

- 作品検索の現在の並び順は `star_count DESC, g.created_at DESC, g.id DESC`。この並び順自体は今回の主目的ではないため、原則維持する。
- 並び順そのものを見直す場合は、別 ADR / 別判断にする。

### 7. seed 増量

検索ページングと総件数表示を検証するため、`src/seed.php` のダミー作品を増やす。

要件:

- 同一タグで 13 件以上ヒットするデータを作る。
- キーワード検索でも複数ページにまたがる語を用意する。
- 公開 / 非公開が混ざる状態を維持する。
- 作者違いが混ざる状態を維持する。
- ログイン中の自分除外を確認できる状態を維持する。
- トップの新着・人気ランキング用の star 傾斜を壊さない。
- seed は冪等にする。

目安:

- 現状の作品1〜24に加えて、検索検証用の公開作品を少なくとも 30 件程度追加する。
- `雨`, `抽象`, `都市`, `空` など、既存タグとつながるタグを意図的に増やす。
- 「雨」「都市」など、タイトル / 説明にも複数ヒットする語を入れる。

### 8. コメント機能は後日に回す

コメント機能は本当は欲しいが、今回の検索改修とは別の追加機能として扱う。

理由:

- 新しい DB テーブルが必要。
- コメント投稿 / 削除 API が必要。
- 表示 UI が必要。
- 所有者削除、投稿者削除、場合によっては通報 / モデレーション方針が必要。
- 今日中の期日に対してスコープが大きい。

今回の優先は、既存検索機能の完成度を上げること。

## 受け入れ条件

- 作品検索で総件数が表示される。
- ユーザー検索で総件数が表示される。
- `10 / 30 / 50` の表示件数切り替えができる。
- 表示件数・ページ番号が URL に残る。
- ページ再読込で同じ検索状態が復元される。
- ページ切り替えで「前へ / 次へ」が正しく有効 / 無効になる。
- 表示件数を切り替えたとき、直前に見ていた先頭カードを含むページへ移動できる。
- タグ検索で 2 ページ目以降を検証できる seed がある。
- キーワード検索で 2 ページ目以降を検証できる seed がある。
- 公開のみ・自分除外・非公開非開示が維持される。

## 想定検証

docs-only では実行不要。実装時の想定:

```sh
docker compose ps
docker compose exec -T app php -l public/api/search.php
node --check public/Script/search.js
docker compose exec -T app php /var/www/html/src/seed.php
curl -s "http://localhost:8080/api/search.php?q=雨&page=1&per_page=10" | head -c 500; echo
curl -s "http://localhost:8080/api/search.php?tag=抽象&page=2&per_page=10" | head -c 500; echo
curl -s "http://localhost:8080/api/search.php?type=users&q=a&page=1&per_page=10" | head -c 500; echo
git diff --check
```

手動確認:

- `search.php?q=雨&page=2&per_page=10` を開いて、2ページ目として表示される。
- 表示件数を 10 から 30 に変えると URL が更新される。
- 表示件数変更後、直前の先頭カードを含むページへ移動する。
- 総件数とページ数が UI と API で矛盾しない。
- ユーザー検索も作品検索と同じ操作モデルで使える。
