# PrismStar — 読みどころ（設計のこだわり・判断軸・技術選定）

PrismStar が **どんな判断軸で設計されているか** をまとめた索引。
個別の意思決定の詳細は `docs/decisions.md`（ADR）、技術仕様は `docs/spec.md`、開発フローは `docs/ai-roles-and-workflow.md` に下りていける。

---

## こだわりポイント

1. **動いているものを作り直さない。** PHP + MySQL + Docker が既に動く `develop` を本体に選び、それを伸ばした。流行りで作り直すのではなく、価値のある資産を活かす判断（[ADR-001](decisions.md) / [ADR-002](decisions.md)）。
2. **「安全な方を選んだ」を理由ごと残す。** GitHub 連携の token はフロントに置かず**サーバ側 PHP の環境変数**に隠す。フロント直叩きとの比較・SSRF 対策・却下理由まで記録し、"なぜ安全か" を説明できる状態にした（[ADR-003](decisions.md)）。
3. **意思決定と理由を記録に残す。** 「何を・なぜ・何を却下したか」を ADR として固定し、設計（方針・handoff）と実装を分けて進める。判断が後から辿れる開発の進め方（[ADR-005](decisions.md) / [ADR-007](decisions.md)）。
4. **名前がコンセプトを体現する。** **PrismStar**＝Prism（あらゆる色＝多様性）＋ Star（目指せスター／お気に入り⭐）。一語で世界観が立ち、機能（多様な発信・スター評価）と直結する（[ADR-006](decisions.md)）。
5. **「全員が発信者」という一本のコンセプト。** 観る人と作る人の境界をなくし、非エンジニアは画像/文字を、エンジニアは GitHub リポジトリを、**同じ場で発信**できる（[ADR-004](decisions.md) / [ADR-008](decisions.md)）。
6. **評価を「スター(⭐)」という一つの通貨に統一。** サイトのお気に入りも GitHub の star も同じ単位で扱い、GitHub 連携を"取ってつけ"でなくコンセプトの中心に溶かした（[ADR-009](decisions.md)）。
7. **守るべき不変条件を明文化して死守。** 認証必須・プリペアドステートメント・所有者限定削除・token 秘匿などを safety invariants として固定し、設計から実装まで崩れないようにした（`docs/ai-roles-and-workflow.md` §4）。
8. **発見の入口を「網羅」と「キュレーション」で分ける。** 検索は探した結果を全部たどる場として「もっと見る」で段階読み込み、トップ（おすすめ）は今の動きを一望する入口として **新着・人気の2軸ランキング**（各トップ5）に。1画面に詰め込まず役割で分けた（[ADR-029](decisions.md)）。

## 判断軸（横断する設計原則）

ADR は「点の決定」、ここはその**底に流れている原則**。新しい判断もこの軸に照らして決める。

| 判断軸 | 中身 | 効いている場面 |
|---|---|---|
| **必要だから使う、無理には使わない** | 技術は流行りでなく必然性で採る／捨てる。 | Node 再実装を却下し PHP を維持。Docker は「環境を再現可能にする」必然があるので採用（[ADR-002](decisions.md)）。 |
| **安全な選択肢を、理由ごと選ぶ** | 安全側に倒すだけでなく、**比較と却下理由**を残して説明可能にする。 | GitHub token をサーバ側 env に隠す（[ADR-003](decisions.md)）。token 漏れ検査の対象定義を見直し（[ADR-011](decisions.md)）。 |
| **動くものを壊さない／作り直さない** | 既存の動作実装を活かし、手戻りと過剰作り込みを避ける。 | `develop` を本体に選択（[ADR-001](decisions.md)）。リファクタは挙動を変えない範囲に限定（[ADR-012](decisions.md)）。 |
| **判断を記録して辿れるようにする** | 決定を会話で流さず、背景・理由・代替案を残す（説明責任）。 | `docs/decisions.md`（ADR-lite）と `docs/*-handoff.md`（[ADR-007](decisions.md)）。 |

## 技術スタックと選定理由

「採用したか」だけでなく **「なぜ採ったか／なぜ他を採らなかったか」** を併記する。一覧表は `docs/spec.md §2`、各判断の詳細は ADR。

| 層 | 採用 | 採った理由（必然性） | 採らなかった代替案と却下理由 |
|---|---|---|---|
| バックエンド | **PHP 8.2 (Apache)** | 既に動く実装があり、セッション認証・CRUD が完成済み。`include` によるサーバ側テンプレートも活かせる。 | **Node + Express で再実装** → 動く実装の価値のない作り直し。「必要だから使う」原則に反するため却下（[ADR-002](decisions.md)）。 |
| DB | **MySQL 8.0** | 既存スキーマ（users / gallery / contacts）が動作。プリペアドステートメント前提の安全な構成。 | 別 DB への移行 → 移行コストに見合う理由がなく却下。 |
| フロント | **HTML / CSS / Vanilla JS（ビルドなし）** | フレームワーク・ビルド工程なしで完結。`<template>` + JS の DOM 生成で十分に動く。 | React 等の SPA 化 → この規模で必然性がなく、学習・運用コストに見合わないため見送り。 |
| インフラ | **Docker Compose（app + db）** | `app`/`db` 2コンテナで開発環境を再現可能にする必然性のある採用。 | ローカル直起動 → 再現性で劣るため却下。 |
| 外部連携 | **GitHub REST API（サーバ側 PHP 経由）** | token をサーバ側 env に隠して認証付き（5000 req/h）で安全に叩ける。SSRF 対策込み。 | **フロント直叩き** → token がソースに丸見え／未認証 60 req/h。安全性で却下（[ADR-003](decisions.md)）。 |

## 設計の難所と解決（苦労した点）

「どこが難しく、どう判断して解いたか」。各判断の詳細は `docs/decisions.md`（ADR）。

1. **GitHub token を安全に扱う。** フロント直叩きは楽だが token がソースに露出（または未認証で 60 req/h）。サーバ側 PHP proxy に隠し、**SSRF 対策**（username を `^[A-Za-z0-9-]+$` に制限）・env 秘匿・配信資産への漏れ検査まで詰めた。「両者を比較して安全側を選んだ」過程ごと残す（[ADR-003](decisions.md) / [ADR-011](decisions.md)）。
2. **動くものを作り直さない見極め。** Node 再実装に流れず、`develop` の PHP 資産が動くことを確認して活かす判断（[ADR-001](decisions.md) / [ADR-002](decisions.md)）。
3. **状態モデルを削ぎ落とす。** 「下書き / 投稿 / 公開・非公開」を並べた末に "下書き＝非公開" と気づき、限定公開の要件がない以上3状態は過剰と判断、**公開/非公開の2状態**へ単純化（[ADR-014](decisions.md)）。過剰設計を避ける判断。
4. **公開と管理を分ける（所有モデル）。** 公開フィード「みんなの作品」の詳細ページに削除があるのは操作モデルとして不適切。管理（編集/削除/公開切替）はマイページに集約し、公開ページは閲覧専用に。所有者限定はサーバ側 `WHERE id=? AND user_id=?` で担保（[ADR-013](decisions.md)）。デモが単一ユーザー所有で誤解を生んだため、seed を複数ユーザー化して所有モデルを可視化。
5. **挙動を変えないリファクタ。** 8ページの PHP 部品化（`.html`→`.php`、header/footer を partial 化）を「挙動不変」で通すため、`php -l` ＋ HTTP スモーク（全ページ 200 / 旧 `.html` 404）で機械的に検証（[ADR-012](decisions.md)）。
6. **「ランキングが順位に見えない」を seed で解く。** おすすめを人気ランキングにしたが、星が無いと全作品が同点で順位が出ない。seed をテスト fixture として **ユーザーを増やし星を降順で傾斜配分**（作者以外から付与・`INSERT IGNORE` で冪等）し、人気順が実データで見えるようにした。あわせて検索の `LIMIT/OFFSET` は **int 化した値だけを SQL に埋める**（PDO プレースホルダの癖を避けつつインジェクションを断つ）。発見系は「検索＝網羅／トップ＝キュレーション」と役割を分ける所が設計の肝だった（[ADR-029](decisions.md)）。
